## 前端交互行为规范

这一块既不属于技术选型，也不属于视觉设计原则，但它是连接二者的胶水层。如果没有把它想清楚，开发过程中每个组件都会各自发明一套行为逻辑，最终产品会显得"割裂"。

------

#### 一、导航模型：工作区是持久的，内容是流动的

平台的左侧边栏不是普通的导航菜单，它是**工作区的持久状态**。用户在会话页生成了一批素材、切换到素材库看了一眼、又回到会话——这整个过程，侧边栏的结构不应该重置，滚动位置不应该丢失。

具体行为：

+ 侧边栏的展开/折叠状态持久化到 localStorage，跨会话保留
+ 当前所在位置的高亮是精确的——不是"我的库"整个板块高亮，而是精确到"我的库 → 某个具体会话"这一项
+ 从素材详情页返回列表时，列表应该恢复到上次的滚动位置和筛选状态，而不是重置到顶部。这在 Next.js App Router 里需要用 `useScrollRestoration` 或手动管理，但它对设计师这种"需要反复对比多个素材"的使用场景至关重要

------

#### 二、加载策略：骨架屏的使用要有节制

骨架屏（Skeleton）被滥用了。它适合**布局已知、内容未知**的情况，但很多场景用它反而制造焦虑。

规则：

+ **素材库列表**：用骨架屏，因为网格布局已知，卡片尺寸固定，骨架屏能稳定页面结构
+ **素材详情页**：不用骨架屏，用渐进式加载——名称和描述先出现（来自列表页的缓存数据），图像区域用占位色块，等图片加载完后淡入
+ **AI 会话**：流式输出，字符逐渐出现，不需要任何 loading 状态
+ **画布**：进入时先渲染空画布（立即可见），素材节点在后台加载完成后逐个淡入，不阻塞用户进入画布的动作

核心原则：**用户应该总是在一个"可以操作的界面"里等待，而不是盯着一个灰色的 loading 界面**。

------

#### 三、拖拽行为的统一约定

平台里有多处拖拽：从库拖入画布、画布内移动节点、图像列表排序。这几种拖拽的**反馈语言必须统一**，否则用户要重新学习每一处。

约定：

+ **拖拽开始**：被拖拽的元素轻微缩小到 95%（不是抬起，缩小），并降低透明度到 70%，产生"正在被拿起"的感觉
+ **拖拽中**：跟随鼠标的幽灵元素保持原始尺寸，不缩小，因为用户需要看清楚自己在拖什么
+ **放置区域**：目标容器用 1px 内发光边框高亮（而不是背景色变化），因为背景色变化会干扰对内容的判断
+ **放置成功**：目标位置的元素用一个短促的弹入动画（100ms，scale 从 0.95 到 1.0）确认落点
+ **放置失败**（拖到了无效区域）：幽灵元素以一个短促的"抖动后归位"动画返回原位，不要静默失败

------

#### 四、键盘行为：设计师是重度键盘用户

目标用户是游戏设计师，他们在 Figma、Photoshop 里有大量键盘操作习惯。平台的键盘支持不是"无障碍合规"，而是**真正的效率工具特性**。

必须支持的键盘行为：

+ 素材库列表里，方向键在卡片间移动焦点，Enter 进入详情，Escape 返回列表
+ 素材详情页里，`E` 直接进入名称编辑模式（不需要点击），`Escape` 确认并退出编辑
+ 画布里，`Space + 拖拽` 是平移（Figma 标准），`Ctrl/Cmd + 滚轮` 是缩放，`Delete` 删除选中节点
+ AI 会话里，`Ctrl/Cmd + Enter` 发送消息，`Shift + Enter` 换行——这是开发者工具的标准，设计师也熟悉
+ 全局 `Ctrl/Cmd + K` 打开命令面板（快速跳转到任意素材、会话、项目）

命令面板是一个优先级很高的功能，因为随着用户的素材库越来越大，"搜索"比"浏览"更高效。

------

#### 五、表单行为：自动保存，而不是提交按钮

平台里的编辑场景（素材名称、描述、注释）不应该有"保存"按钮。设计工具里没有人会手动点保存。

规则：

+ **自动保存**：用户停止输入 800ms 后自动触发保存，状态栏显示"已保存"的细节提示
+ **保存状态的呈现**：不用 toast，用页面顶部或侧边的一个极小的状态指示器（类似 Notion 右上角的"正在保存…"/"已保存"），不打断视线
+ **冲突处理**：如果同一素材在另一个标签页被修改了，当前页在下次聚焦时出现一个 inline 提示（"检测到外部修改，是否刷新"），而不是静默覆盖
+ **唯一的例外**："公开素材"操作必须显式确认，因为它不可逆——这是前面设计原则里提到的"高风险操作需要阻力"
+ **显式操作的成功反馈**：与自动保存不同，用户在对话框中确认后的不可逆或重要操作（例如「发布到探索」「复制到私库」）成功后，使用全局 toast（sonner）给出简短成功提示，避免用户不确定是否已生效

------

#### 六、错误边界的分级处理

错误不应该都触发同样的处理方式。按照影响范围分三级：

+ **局部错误**（某张图片加载失败、某次 AI 请求超时）：在出错的组件内部处理，显示一个内联的重试按钮，不影响页面其他部分。这是最常见的情况，绝对不要因为一张图加载失败就让整个页面崩溃
+ **功能级错误**（生图服务不可用、会话无法创建）：在功能入口处显示一个 banner 或 callout，说明哪个功能暂时不可用，其他功能正常可用
+ **页面级错误**（路由对应的资源不存在、权限不足）：显示一个完整的错误页，但要提供有意义的出口——"返回我的库"，而不是只有一个冷冰冰的 404

------

#### 七、响应式策略：桌面优先，但平板要能用

平台是专业工具，手机端不在考虑范围内。但平板（iPad Pro 这类）是设计师在会议室、咖啡馆的常见设备，尤其是浏览素材库和查看情绪板这两个场景。

策略：

+ **桌面（1280px+）**：完整的双栏或三栏布局，侧边栏展开
+ **平板（768px–1280px）**：侧边栏默认折叠为图标模式（只显示图标，hover 展开），主内容区全宽。素材网格从 4 列降为 2 列
+ **画布页面**：在平板上完全可用，tldraw 本身支持触控，需要确认所有自定义节点也支持触摸事件
+ **低于 768px**：不做响应式适配，但要显示一个友好的"请在桌面端使用"的提示，而不是让布局直接崩掉

------

#### 八、性能预算：用户感知的阈值

最后，给几个必须守住的性能指标，这些数字不是为了 Lighthouse 评分，而是为了保住设计师对工具的信任感——**一个卡顿的设计工具会让用户立刻质疑它的专业性**。

+ 素材库列表首屏渲染：< 1.5 秒（LCP）
+ 从列表进入素材详情的转场：< 200ms（使用 Next.js 的 prefetch，鼠标 hover 卡片时就开始预取）
+ AI 会话的首字符出现延迟：< 800ms（流式输出，不等完整响应）
+ 画布打开并可交互：< 2 秒（画布节点可以异步加载，但画布本身必须立即可操作）
+ 图像懒加载：所有图像使用 `loading="lazy"` + IntersectionObserver，不在视口内的图片不加载

图像是平台最重的资源。需要在后端约定好一套图像规格——卡片缩略图（400px 宽）、详情页中图（800px 宽）、原图——CDN 按需裁切，前端只请求当前场景所需的尺寸。
