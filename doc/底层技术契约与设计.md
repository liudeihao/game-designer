### 一、数据结构与 API 契约

#### 核心 Schema

素材（Asset）是整个平台的原子单位，它的 TypeScript 类型需要覆盖所有状态的并集：

```typescript
type Asset = {
  id: string
  name: string
  description: string
  annotation: string | null       // 用户注释，不参与生图
  authorId: string
  author?: { id: string; username: string; displayName: string | null }  // 非 deleted 时由 API 联表返回（与 UserPublic 一致）
  createdAt: string               // ISO 8601
  updatedAt: string

  visibility: 'private' | 'public' | 'deleted'
  forkedFromId: string | null     // null 表示原创
  forkCount: number               // 只做展示，不做排序依据

  images: AssetImage[]
  coverImageId: string | null     // 选定的展示图
  groupId: string | null         // 仅私库分组；public 恒为 null（公开即移出私库分组）

  // 仅在 deleted 状态下有意义
  deletedAt: string | null
}

type AssetImage = {
  id: string
  assetId: string
  url: string                     // CDN 地址，后端负责多尺寸
  extraPrompt: string | null      // 生图时的附加描述
  createdAt: string
  generationStatus: 'pending' | 'done' | 'failed'
}
```

几个需要和后端约定的细节：

+ `visibility: 'deleted'` 的素材，后端返回时只保留 `id`、`forkedFromId`、`visibility`、`deletedAt` 字段，其余字段置空或省略。前端用这个状态渲染"幽灵节点"
+ `groupId` 只用于作者的**私有**素材在「我的库」里的分组。`public` 素材不占用私库分组（`group_id` 恒为 `null`）：公开后相当于分享/归档；若需生图、分组或深度迭代，用户应在私库中 fork/复制为新的私有素材。`PATCH` 不可为已公开素材设置 `groupId`。
+ **我的库列表**：作者可对**仅自己可见**（`visibility: private`）的素材使用「批量管理」多选，经确认后调用 `DELETE /api/assets/{id}` **软删除**（`visibility → deleted`，列表不再展示）。**已公开**素材不可通过该接口删除（接口返回 400）；全站「探索」页不受此操作影响。
+ **素材标签**：每用户独立命名空间（`tags` / `asset_tags`）；`GET /api/asset-tags` 返回各标签素材计数（侧栏）；详情响应包含 `tags: string[]`；仅私有素材可通过 `PATCH /api/assets/{id}` body `tags: string[]` 覆盖绑定。
+ **素材列表**：`GET /api/assets`（私库或探索）支持查询参数 `sort`（`created_desc`|`updated_desc`|`fork_desc`，分页游标随排序字段变化）、`q`（名称 / 描述 / 备注模糊匹配）、`tagId`（私库）、`hasImage=true` 或 `img=yes`（仅有图）、`img=no`（无图）。
+ 图像的 URL 不返回原图地址，返回带尺寸参数的 CDN 模板，例如 `https://cdn.example.com/images/{id}?w={width}`，前端按场景拼接宽度参数
+ `forkCount` 除详情衍生标记外，私库列表可选用「分叉次数」排序（`sort=fork_desc`）

#### Fork 树的加载策略

Fork 树不应该一次性返回整棵树。平台里素材可以无限层级地衍生，一次性加载会有性能问题，而且大多数时候用户只关心"我 fork 自谁"和"谁 fork 了我"这两层。

约定的加载策略：**局部展开，按需懒加载**。

接口设计为：

```
GET /assets/{id}/forks?direction=upstream&depth=1
GET /assets/{id}/forks?direction=downstream&depth=1
```

前端在 Fork 树可视化组件里，初始只加载当前节点的直接上游（1 级）和直接下游（1 级），渲染成一个三层的小图。用户点击某个上游节点的"展开"时，再发请求获取那个节点的上游，以此类推。`depth` 参数预留，但 v1 阶段 API 只实现 `depth=1`。

`forks` 响应里每个 `ForkNode` 在**当前访问者可读**且非删除时，可带 `coverImageUrl`（封面或首张图 URL，与 `fork-graph` 节点一致），便于详情页等场景展示缩略图；不可读的他人私有素材不返回名称与封面。

#### 分页策略

列表接口统一使用**游标分页**，不用页码。理由是砖墙布局的无限滚动天然适配游标分页，而且当用户在浏览过程中有新素材被插入时，游标分页不会导致重复加载。

```
GET /assets?scope=public|private&cursor={lastId}&limit=24&sort=createdAt_desc
```

公开列表可选 `authorUsername={username}`（仅与 `scope=public` 合用），用于用户主页 `/u/{username}` 只展示该用户已发布素材。`GET /api/users/{username}` 返回公开资料：`displayName`、`avatarUrl`、`coverUrl`（可空，http(s) 直链）、`stats`（`publicAssets`、`forksReceived` 为公开素材的 `fork_count` 之和、`projects`）；`PATCH /api/me` 可部分更新 `displayName` / `avatarUrl` / `coverUrl`（空或 null 清空）。`users` 表字段 `avatar_url`、`cover_url`。

响应体结构：

```typescript
type PaginatedAssets = {
  items: Asset[]
  nextCursor: string | null   // null 表示已到末尾
  total: number | null        // 可选，用于显示"共 N 个素材"
}
```

前端用 TanStack Query 的 `useInfiniteQuery` 消费这个接口，当用户滚动到列表底部时自动触发下一页请求。

### 二、状态管理边界

#### 画布与素材库的数据同步

画布节点**只保存 Asset ID，不深拷贝数据**。这是最重要的一个架构决策，必须在开发开始前定死。

具体实现逻辑：tldraw 的自定义节点类型存储的数据结构如下：

```typescript
type AssetNodeData = {
  type: 'asset-node'
  assetId: string
  // 画布专属属性，与素材数据无关
  lockedOnCanvas: boolean
}
```

当画布渲染这个节点时，从 TanStack Query 的缓存里根据 `assetId` 取数据。如果缓存里有就直接用，缓存失效时会自动重新请求。这样，用户在素材库里修改了素材名称，画布里对应的节点在下次数据刷新时自然反映最新状态，不需要额外的同步机制。

这个设计还带来一个好处：画布的持久化数据（存到后端的那份）非常轻量，只是坐标和 ID 的映射，不包含素材内容本身。

#### 服务端组件与客户端边界

明确划分如下：

**服务端渲染（Server Components，无 `'use client'`）：**

+ `/explore` 公开素材库列表页——SEO 最重要的页面，首屏必须是服务端渲染的 HTML
+ `/u/[username]` 用户主页——同样有 SEO 需求
+ 素材详情页的静态部分（名称、描述、作者信息）——用 Next.js 的 `generateStaticParams` 对公开素材做静态生成

**客户端组件（`'use client'`）：**

+ 砖墙网格的无限滚动容器——因为 IntersectionObserver 依赖浏览器 API
+ 所有 Framer Motion 动画组件
+ AI 会话页面——完全客户端，依赖流式 fetch 和实时 state
+ tldraw 画布——整个画布页面都是客户端组件，Next.js 的动态导入配合 `ssr: false`
+ 素材详情页的图像管理区——因为有生图状态轮询
+ 工作区 **主内容区**（`WorkspaceShell` 的 `main` 内）若干 **横向分栏**（列表侧栏与主区、会话「聊天 | 暂存」、我的库侧栏、素材详情「文案 | 图带」等）使用 `react-resizable-panels`：用户可拖动分隔条调整比例，布局持久化在 **localStorage**（键名带前缀 `layout:`，如 `layout:sessions-list-sidebar`、`layout:session-chat-drafts`、`layout:staging-group-settings-drafts`、`layout:session-chat-composer`（会话内聊天记录区与输入区**纵向比例**，中间**可拖动**横条调整；分隔条为 **上下内边距加宽命中区 + 中心 1px 线**，便于鼠标抓取）、`layout:library-sidebar`、`layout:explore-preview`（探索页网格与右侧预览面板）、`layout:asset-detail`）。**全局左侧图标轨与顶栏**不参与分栏，尺寸固定。`main` 自身 `overflow-hidden`，其内一层 `overflow-y-auto` 供非填满型页面滚动；**会话页**中消息历史仅在聊天上面板内滚动，输入区固定在下方面板；**暂存侧栏**仅条目列表区域滚动，标题与「加入暂存」表单固定在列表上方。会话详情内 **聊天区 | 暂存区** 横向 `Group` 与列表 | 主区一致：`h-full`/`min-h-0` 贯通高度链、`Panel` `overflow-hidden`、分隔条 `self-stretch` 与 `divider` 色，使竖向分隔线与列表右侧边线一样纵贯主内容区（顶栏以下）。可滚动区域使用 `globals.css` 中的 `.gd-scrollbar`（及全局 `textarea` 规则）：轨道与滑块用 `color-mix` 引用 `--accent` / `--text-primary`，与主题一致。

**混合模式（Server Component 包裹 Client Component）：**

+ 素材详情页整体。外层是 Server Component 负责 fetch 初始数据并注入，内层的编辑区、图像区是 Client Component 接管交互。这样首屏渲染不需要等待客户端 JS 加载完成

**Zustand 的职责**只用于纯客户端的 UI 状态，例如侧边栏的折叠状态、命令面板的开关状态、画布的视图偏好设置。一切涉及服务端数据的状态，都交给 TanStack Query 管理，Zustand 不持有业务数据。

**会话详情与聊天历史**：`SessionWorkspace` 对 `["session", id]` 使用 `useQuery`，RSC 注入的 `initialData` 仅作首屏占位；因 TanStack Query 在**已有缓存**时不会用新的 `initialData` 覆盖，故配置 `initialDataUpdatedAt: 0`、`staleTime: 0`、`refetchOnMount: "always"`，保证每次进入会话都会向服务端拉取最新消息列表。流式回复在服务端落库完成后才与列表一致，故在流正常结束后 **`await refetch()`** 再收起流式 UI，避免「必须整页刷新才看到助手消息」的竞态。

**空会话在离开页时清理**（例如从 `/library/sessions/new` 点「开始」后立刻进详情却未发聊）：在 `/library/sessions/{id}` 中，若用户**从未产生用户侧聊天消息**就离开（关闭页或路由离开），`SessionWorkspace` 在卸载时通过 `useDiscardEmptySessionOnLeave` 调用 `DELETE /api/sessions/{id}`，避免侧栏里堆积无意义「壳」会话；**独立/未分组暂存**下若已有手填「暂存」行，则不自动删除；**组内共享暂存**下仅以「是否已有用户侧聊天消息」为准（组内可能已有他会话的暂存，不能用条数当判据）。`request` 在途（`streaming`）时不删。实现上用 `setTimeout(0)` 推迟一次删除，并配合模块级 `Map` 在 React Strict 模式重挂载时撤消上次卸载挂起的任务。

**会话内「聊天 | 暂存」分栏**：**`md`（768px）及以上**为左右分栏，使用 `WorkspaceHorizontalSplit`（`layout:session-chat-drafts`）；**窄屏**为上下分栏，使用 `WorkspaceVerticalSplit`（`layout:session-chat-drafts-stack`），分隔条同样可拖动以调整上下比例（不再是仅 `border-t` 的固定网格）。`Group` 需同时传入 `useDefaultLayout` 的 **`onLayoutChange` 与 `onLayoutChanged`**，且横向时 **右侧面板**需有 **`defaultSize`**（与左侧合计 100%），否则易出现布局不同步、拖动不生效等问题。分隔条视觉为 **1px**（`bg-divider`），`resizeTargetMinimumSize` 保障拖动命中；`z-20` 与列表等区域边线一致；会话列表与主区的侧栏分栏仍在 **`lg`** 断点与主区并排。

**会话主区与 Shell 滚动**：`/library/sessions/{id}`、分组详情、**`/projects/{id}/...` 下除列表与 `/new` 外的子路由**等 **填满视口** 的路由在 `WorkspaceShell` 主内容容器上使用 **`overflow-hidden`**，由聊天列表、输入区分栏（`WorkspaceVerticalSplit`，`layout:session-chat-composer` 或项目设计内 `layout:chat-composer:{sessionId}`）与暂存区（仅素材会话）各自内部滚动；避免 Shell 层 `overflow-y-auto` 把整块工作区滚走、输入框离开视口。列表页、`/new`、`/archive` 仍为 Shell 层纵向滚动。

**游戏项目（设计 + 情绪板）**：后端 `projects` 表存 `name`、`canvas_document`（JSON）与 **`design_document`（TEXT，Markdown 游戏设计文档，默认空）**。**设计会话**：`chat_sessions.project_id` 指向项目时可有多条线程；**全局** `GET /api/sessions` **仅返回** `project_id IS NULL` 的素材向会话。`GET/POST /api/projects/{id}/sessions` 列出/创建项目内设计会话；`POST/DELETE /api/projects/{id}/assets` 维护 **`project_assets`**（仅作者私有、未删除素材可链接），`GET /api/projects/{id}` 的 ### 三、AI 流式交互的解析协议

这是技术难度最高的一块，需要前后端约定一套明确的流格式。

#### 流格式约定

推荐使用**带类型标记的 JSON Lines 格式**，每一行是一个独立的 JSON 对象，以换行符 `\n` 分隔：

```
{"type":"text","delta":"我来帮你设计一个"}
{"type":"text","delta":"赛博朋克风格的垃圾桶。"}
{"type":"asset_start","id":"temp_1"}
{"type":"asset_field","id":"temp_1","field":"name","delta":"废弃的自动贩卖机"}
{"type":"asset_field","id":"temp_1","field":"description","delta":"一台锈迹斑斑的"}
{"type":"asset_field","id":"temp_1","field":"description","delta":"全息广告贩卖机，"}
{"type":"asset_end","id":"temp_1"}
{"type":"text","delta":"这个素材适合作为场景道具。"}
```

前端维护一个流解析器，处理这些事件类型：

+ `text`：追加到当前聊天气泡的文本内容，产生打字机效果
+ （可选/未来）`asset_start` + `asset_field` + `asset_end`：在聊天流中原地渲染"正在生成中的素材卡片"骨架；`asset_end` 时完成态。若接入会在流中直出成稿的 Model，可启用该路径
+ **会话「暂存」的当前产品规则**：`draft_assets` 以用户**手填**的**名称**与**视觉描述**（画面、色光、材质、氛围等外观向信息，而非玩法/系统说明）为准，经 `POST /api/sessions/{sessionId}/drafts` 写入；**聊天流不自动**创建暂存行。暂存条目可在同一会话内 **`PATCH` / `DELETE`**（路径带 `tempId`）编辑或删除。**导出到「我的库」**使用 `POST /api/sessions/{sessionId}/drafts/{tempId}/export-to-library`：在**单事务**中插入私有 `assets` 行并**删除**对应暂存行（避免重复点击产生多条库内素材）；`sessionId` 的语义与 `PATCH/DELETE` 草稿相同（共享组暂存时任选组内一会话 id）。前端在导出前须**二次确认**，失败时展示错误信息。**单条**导出成功后，确认弹窗不立即关闭，改为展示成功说明及 **`/library/assets/{id}`** 的链接（新标签页打开详情）；用户关闭弹窗后回到会话。**批量 / 全部导出**仍保持确认后关闭、无详情链接。
+ **会话分组与暂存范围**：用户可建立「**会话分组**」（与素材库文件夹 `asset_groups` 不同，独立表与 API），并为分组设置 **`draft_staging`**：`independent` 时组内**每个**会话各有一套 `draft_assets`（行挂 `session_id`）；`shared` 时**整组**共用一套暂存（行挂 `group_id`）。`GET/POST` `/api/session-staging-groups` 建组；`GET` `/api/session-staging-groups/{groupId}/drafts` 返回该组**全部**暂存行（`shared` 为组级池；`independent` 为组内各会话行的合并，项上带 `ownerSessionId` / `ownerSessionTitle` 以便 `PATCH/DELETE` 仍走 `/api/sessions/{sessionId}/drafts/{tempId}`）；`PATCH/DELETE` `/api/session-staging-groups/{groupId}` 改名或改模式（切换模式时若会冲突，接口返回 400 要求先清空相关暂存）；`PATCH` `/api/sessions/{sessionId}` 可设 `stagingGroupId` 将会话移入/移出分组。列表/详情在 `stagingGroup` 上返回分组的 `id`、`name` 与 `draftStaging`；`draftAssetCount` 在 `shared` 下按**组**计数，在 `independent` 下按**当前会话**计数。**界面**：侧栏中「**会话**」与「**分组**」为**同级**区块：「会话」下列 **新会话** 与**全部未分组**会话；「分组」标题行右侧 **+** 下拉创建分组（名称与暂存模式）；侧栏不设「全部」入口。桌面端无额外「会话库」顶条；移动端顶条仅「← 会话」与「新会话」。**分组独立页**路由为 `/library/sessions/groups/{groupId}`：侧栏点击分组名称进入；**主区**为横向分栏（`layout:staging-group-settings-drafts`）：**左**为分组名称/暂存模式/组内会话列表/删除分组等，**右**为暂存区，内容同该组 `GET .../drafts`（独立模式为组内合并；共享模式为组池）。可在此新增/编辑/导出（`POST` 仍经 `sessionId`；组内多会话时 UI 可选「新增暂存归属会话」）；新建会话可带 `?group=` 预选分组。会话库侧栏中分组为**可展开的一行**（下列所属会话）；列表行右侧**铅笔**与**垃圾桶**默认隐藏，仅在**当前选中**（会话行为当前打开的会话；分组行**仅**在打开该分组页 `/groups/{id}` 时算选中，打开组内某会话时不算）、**鼠标悬停该行**或**行内键盘聚焦**时显示；**铅笔**内联重命名（`PATCH /api/sessions/{id}` 的 `title`、`PATCH /api/session-staging-groups/{groupId}` 的 `name`），**垃圾桶**删除会话（`DELETE /api/sessions/{id}`，库表对消息与挂 `session_id` 的 `draft_assets` 级联删除，确认框须说明未入库暂存会一并删除；**组内共享暂存池**不因删除单条会话而整体清空），Enter/Esc 同前。**会话详情页**布局与是否入组无关：左栏始终为聊天；标题旁设置仅调整「所属分组」（侧栏归类，分组的名称与暂存模式在分组页管理）。右栏暂存即 `GET /api/sessions/{id}` 的 `draftAssets`：`independent` 为当前会话条目，`shared` 为该组共享池；`POST` 新暂存仍走当前 `sessionId`。

这套协议的好处是**对 SSE 和 WebSocket 都兼容**，流格式与传输方式解耦。v1 阶段用 SSE，后续如果需要双向实时通信（例如多人协作画布）切换到 WebSocket 时，流格式不需要改变。

#### 后端 LLM（Go）

+ **驱动**：环境变量 `AI_DRIVER` 为 `mock`（默认）或 **`openai`**（兼容旧值 `eino`，语义相同）。聊天与生图均使用官方 **`github.com/openai/openai-go`** 调用 OpenAI 兼容 HTTP API；密钥与模型仅在后端配置。
+ **常用变量**：`AI_API_KEY`（`openai` 必填）、`AI_BASE_URL`（可选，兼容代理/Azure 端点）、`AI_CHAT_MODEL`（可选，默认 `gpt-4o-mini`）、`AI_HTTP_TIMEOUT`（可选，默认约 90s）。详见仓库根目录 `.env.example`。
+ **上下文**：每次用户发消息后，后端从 `chat_messages` 读取多轮历史（有条数上限）并拼一条 **system** 说明（含项目关联素材摘要、项目 **`design_document` Markdown** 截断片段及产品规则）；流式输出结束后将**完整助手文本**落库为一条 `assistant` 消息。
+ **生图**：`POST /api/assets/{id}/images` 在 `AI_DRIVER=openai`（或别名 `eino`）且图像客户端初始化成功时，调用 **OpenAI Images API**（同一 SDK）；与聊天共用 `AI_API_KEY` / `AI_BASE_URL`。默认 `AI_IMAGE_MODEL=dall-e-3`、`AI_IMAGE_SIZE=1024x1024`；`dall-e-2`/`dall-e-3` 请求 `response_format=b64_json`；`gpt-image-*` 默认返回 b64。入库为 **`data:<mime>;base64,...`**（按响应 `output_format` 选择 `image/png` 等），避免临时 URL 过期。前端对 `data:` URL **不再**拼接 `?w=`（见 `imageDisplaySrc`）。

#### 前端解析器的边界情况处理

需要处理的边界：

+ 流中断（网络断开）：记录最后一个 `temp_id` 的状态，提供"重新生成"按钮，不丢失之前已生成的素材
+ 同一条流里出现多个 `asset_start`：解析器用 `id` 区分，各自独立渲染，支持 AI 一次生成多个素材
+ 纯文本回复（AI 在回答问题而不是生成素材）：没有 `asset_start` 事件，只有 `text` 事件，正常渲染聊天气泡

### 四、鉴权与路由守卫

#### 鉴权状态管理

采用**服务端 Session + HttpOnly Cookie** 方案，而不是把 JWT 存在 localStorage。理由是平台有公开内容（SEO 需求），服务端渲染时需要在服务端就能读取鉴权状态，HttpOnly Cookie 天然支持这一点，而 localStorage 只在浏览器端可访问。

具体实现：Go 后端在登录时 Set-Cookie，Next.js 的 middleware 在每次请求时读取 Cookie 验证 Session。前端不存储任何 token 字符串，鉴权状态的唯一来源是 Cookie。

客户端需要知道"当前用户是谁"来渲染 UI 差异（例如是否显示"编辑"按钮），这部分通过一个专门的 `/api/me` 接口在应用初始化时获取，结果存入 TanStack Query 的缓存，作为全局可访问的用户上下文。

#### 路由守卫的分级处理

分三层拦截，每层处理不同的场景：

**第一层：Next.js Middleware（最外层，性能最好）**

在 `middleware.ts` 里处理粗粒度的鉴权拦截：

+ 未登录用户访问任何 `/library/*` 或 `/projects/*` 路由，直接 302 重定向到登录页
+ 已登录用户访问登录页，重定向到 `/library`
+ Middleware 运行在 Edge Runtime，不涉及数据库，只做 Cookie 是否存在的判断

**第二层：Server Component 数据层（细粒度权限）**

当用户访问 `/projects/[id]/canvas` 时，Server Component 在服务端 fetch 项目数据，如果后端返回 403，Server Component 直接渲染权限错误的占位组件，而不是重定向。

重定向和错误展示的选择原则：**如果用户"不应该知道这个资源存在"就重定向到 404，如果用户"知道存在但没有权限"就展示 403 组件**。对于私有项目，未授权用户应该看到 404，而不是 403（不泄露资源存在性）。

**第三层：客户端乐观保护（防止 UI 暴露）**

对于权限敏感的操作按钮（例如"删除公开素材"），不是等用户点击后才从后端拿到 403 再报错，而是在渲染时就根据当前用户身份判断是否显示这个按钮。这层判断依赖 `/api/me` 的数据，是纯 UI 层的保护，**不能代替后端的权限校验**，但能避免用户看到点了没用的操作按钮。

### 下一步行动顺序

建议的推进顺序是：

首先，前后端共同制定 **OpenAPI 规范文档**，把上面的 Schema 和接口格式写成正式契约。这一步完成之前，前端不开始写任何涉及数据的组件，只做纯视觉的 UI 组件库。

其次，用 **MSW（Mock Service Worker）** 在前端本地 mock 所有 API 接口，包括 SSE 流。这样前端开发不依赖后端进度，两端可以并行推进。MSW 的 mock 数据同时也是前端的集成测试数据，一份投入两处用。

然后，按照功能阶段顺序交付：UI 组件库 → 公开素材库（SEO 优先）→ 我的库 + 素材详情 → AI 会话 → 生图工作台 → 画布。每个阶段都有独立的可演示产物，不要等到所有功能都完成才集成。
